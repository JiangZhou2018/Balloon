## 1. 开发目标
- 在 2~4 周内交付一个可试玩、可验证核心玩法的竖屏 H5 MVP。
- 优先验证首局体验和“失败后再来一局”的成长闭环。
- 保证功能范围稳定，不让长线系统稀释核心玩法开发资源。

## 2. 交付范围
- 可运行的 React + TypeScript + Vite H5 工程。
- 1 套完整首局到失败的玩法闭环。
- 基础工坊成长和本地存档。
- 基础数据埋点。
- 基础测试和真机验证结果。

## 3. 优先级

### 3.1 P0 必做
- 工程初始化与基础路由
- 战斗主循环
- 宝箱/气球实体和状态机
- 自动攻击与目标锁定
- Cash/Coins 产出
- 局内升级
- 合约保险与失败结算
- 工坊成长
- 本地存档

### 3.2 P1 强烈建议
- 快逃走优先 / 高价值优先
- 点击集火
- UFO 高价值目标
- 首局轻引导
- 基础音效和特效
- 埋点服务

### 3.3 P2 可延期
- 更多气球机制
- 更丰富结算页
- 更多视觉打磨
- 假广告入口占位

## 4. 页面线框说明

### 4.1 首页
- 顶部：Logo + 当前 Coins
- 中部：主视觉图 + “开始游戏”大按钮
- 底部：工坊、设置

### 4.2 战斗页
- 顶部：波次、保险进度条、Cash
- 中部：天空战斗区域、逃逸线、宝箱、气球、特效
- 右侧：优先级切换
- 底部：10 个局内升级按钮，支持横向滚动或两行布局

### 4.3 结算页
- 中部卡片：本局波次、坠落数、漏箱数、获得 Coins
- 底部按钮：再来一局、前往工坊

### 4.4 工坊页
- 顶部：总 Coins
- 中部：6 个升级项卡片
- 底部：返回首页 / 直接开始

## 5. 前 15 波开发建议
- Wave 1：单个小现金箱，普通气球，确保玩家看懂玩法。
- Wave 2~3：提高 Cash 产出，鼓励首次升级。
- Wave 4~5：加入普通宝箱与金气球，提升收益反馈。
- Wave 6~8：加入富人保险箱，制造高价值选择。
- Wave 9~10：加入第一次漏箱风险提示。
- Wave 11~12：引入钢壳气球，要求玩家理解伤害成长。
- Wave 13~14：引入加速或护盾气球，增加压力。
- Wave 15：出现第一只 UFO 大奖，形成阶段高潮。

## 6. 2 周排期

### 第 1 周
- Day 1：初始化工程、路由、基础页面框架、配置表骨架
- Day 2：实现战斗循环、实体更新、位置与逃逸判定
- Day 3：实现自动攻击、目标锁定、伤害和气球击破
- Day 4：实现宝箱坠落、资源发放、保险扣减、失败结算
- Day 5：实现局内升级系统和战斗 UI
- Day 6：实现工坊成长和本地存档
- Day 7：打通首页 -> 战斗 -> 结算 -> 工坊完整流程

### 第 2 周
- Day 8：补 5 种宝箱和 5 种气球
- Day 9：补目标优先级与点击集火
- Day 10：补 UFO 高价值目标与首局节奏配置
- Day 11：补轻引导、Toast、基础埋点
- Day 12：补基础特效、音效、动效反馈
- Day 13：真机测试、性能优化、修复关键 Bug
- Day 14：完成可试玩版本并整理测试结论

## 7. 4 周排期

### 第 1 周
- 完成 P0 核心闭环

### 第 2 周
- 完成 P1 功能和前 15 波调优

### 第 3 周
- 强化首局引导、战斗表现、性能优化、兼容性修复

### 第 4 周
- 完成小范围试玩版、补测试、补埋点核查、输出版本结论

## 8. 人员分工建议
| 角色 | 任务 |
|------|------|
| 前端主程 | 工程搭建、战斗逻辑、状态管理、页面实现 |
| UI/视觉 | 页面线框、图标、按钮样式、关键反馈特效资源 |
| 测试 | 首局流程验证、机型兼容、存档验证、性能验证 |

## 9. 验收标准
- 用户 5 秒内理解“打气球让宝箱掉落”的核心玩法
- 用户 15~25 秒内获得第一次坠箱爽点
- 用户首局内至少发生一次升级点击
- 用户首局可以完整失败并进入结算
- 工坊升级后再次开局可以明显感知变强
- 连续游玩 3 局无严重阻塞 Bug

## 10. 开发阶段注意事项
- 任何新增需求先判断是否会影响首局闭环验证。
- 若范围膨胀，优先保留 P0 功能。
- 代码按模块拆分，小文件优先，避免单文件过大。
- 所有数值尽量配置化，避免硬编码。
